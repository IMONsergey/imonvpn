# 13. Текущая фаза проекта

## Зафиксированные ограничения

На этой фазе проекта осознанно **не используем**:

- домен
- второй VPS

## Что используем сейчас

- один текущий VPS в Европе
- `Xray Reality` как уже работающий слой для `Hiddify`
- `AmneziaWG 2.0` как уже поднятый нативный слой для `AmneziaVPN`

## Практическая цель этой фазы

Сделать максимально устойчивую систему **в рамках одного сервера**, чтобы:

- `Hiddify` продолжал работать на совместимых устройствах
- `AmneziaVPN` работал на всех пользовательских устройствах
- доступы выдавались раздельно по устройствам
- сервер оставался пригодным для последующего расширения

## Что это значит на практике

### Для Hiddify

- оставляем текущий рабочий `Reality`
- делаем отдельные профили по устройствам
- не используем один общий профиль как постоянную production-схему

### Для Amnezia

- используем уже поднятый `AmneziaWG 2.0` на текущем сервере
- используем `guest access` на отдельное устройство
- `full access` не раздаём, кроме личного управляющего устройства владельца сервера

## Чего эта фаза не решает полностью

Без домена и без второго VPS мы не получаем максимум инфраструктурной устойчивости.

Остаются ограничения:

- один внешний IP
- один провайдер VPS
- нет отдельного доменного слоя маскировки
- нет горячего failover на другой сервер

## Почему всё равно имеет смысл делать именно так

Даже без домена и второго VPS схема:

- `Reality` для `Hiddify`
- `AmneziaWG 2.0` для `Amnezia`

уже даёт два разных клиентских и транспортных пути на одном сервере. Это заметно устойчивее, чем пытаться жить на одном-единственном профиле.

## Текущий блокер от 2026-07-16

Текущий VPS `imonvpn-01` у AlphaVPS находится в состоянии `Disabled` / `Virtual server suspended`.

Проверено:

- service id: `48528`
- IP: `13.140.29.192`
- биллинг сервиса: `Active`
- инвойсы `139774` и `143587`: `Paid`
- `Boot` из панели AlphaVPS не запускает VM и возвращает `Virtual server suspended`
- создан тикет в поддержку AlphaVPS: `#206441`

AlphaVPS ответили в тикете `#206441`: suspension автоматический, причина - исчерпан месячный bandwidth.

Варианты восстановления от провайдера:

- ждать 1-го числа, когда месячный bandwidth counter сбросится автоматически
- купить разовый bandwidth addon: `3 EUR` за `1 TB`
- постоянно увеличить bandwidth: `2 EUR` за `1 TB` в месяц

До снятия suspension на стороне провайдера нельзя объективно отлаживать `AmneziaWG`, `Xray Reality`, `iptables` или клиентские конфиги: серверная VM не запускается штатно.

## Следующий практический шаг

Следующий шаг проекта на этой фазе теперь такой:

1. Выбрать bandwidth-вариант: ждать 1-го числа, купить разовый addon или увеличить тарифный bandwidth
2. После включения VPS проверить `ssh`, `xray-reality`, `nginx`, `docker`, `amnezia-awg2`, `iptables`
3. Проверить подключение `Hiddify` и `AmneziaVPN` с клиентских устройств
4. После восстановления раздать готовые `.conf` по устройствам через `AmneziaVPN`
5. Зафиксировать, какие устройства сидят на `Hiddify`, а какие на `Amnezia`
6. При необходимости добавить отдельный backup-канал под домен
7. После этого думать о втором VPS и failover

## Что уже сделано для Amnezia

- `AmneziaWG 2.0` поднят на текущем сервере
- контейнер работает на `UDP 443`
- выпущен `main.conf`
- выпущены `guest01.conf` ... `guest10.conf`
- клиентские конфиги переведены на `IPv4-only`, чтобы убрать лишний `::/0` маршрут на текущем этапе

`Xray / VLESS Reality`-импорт для `AmneziaVPN` остаётся только как fallback, а не как основной путь.
