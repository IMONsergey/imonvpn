# 13. Текущая фаза проекта

## Решение текущей фазы

После инцидента с bandwidth на старом VPS проект переведён на новый clean-сервер `imonvpn-02`.

Текущая фаза: один VPS, без домена, без второго сервера, без Xray fallback.

## Почему убрали Xray

На старом `imonvpn-01` Xray Reality был оставлен как резервный TCP-слой. После инцидента выяснилось, что высокий расход трафика шёл именно через TCP/Xray-плоскость, а не через активный AmneziaWG-путь пользователя.

Вывод для этой фазы:

- Xray не поднимать
- TCP `443/8443/2053/80` держать закрытыми
- использовать только AmneziaWG на `UDP/443`
- раздавать отдельный peer на каждое устройство/гостя
- смотреть расход по peer, а не гадать по общему bandwidth

## Текущий уровень защиты

- SSH переведён на ключи, password-login отключён.
- `nftables` разрешает только `22/tcp`, `443/udp`, ICMP и established/related.
- `nftables` делает persistent NAT для `10.9.1.0/24`.
- `fail2ban` защищает `sshd` через systemd journal backend.
- `vnstat` собирает общий расход интерфейса.
- `imonvpn-awg-traffic.timer` каждые 5 минут собирает per-peer счётчики из `awg show`.

## Следующий практический шаг

1. Перенести устройства пользователя на новые профили `83.171.203.63:443/udp`.
2. Старые профили `13.140.29.192` больше не использовать.
3. Через 1-2 дня проверить отчёт `/usr/local/sbin/imonvpn-awg-report --report-only`.
4. Если один guest начнёт резко жечь bandwidth, отозвать именно его peer, а не весь сервер.
5. Если РФ-блокировки начнут мешать UDP/AWG, следующий технически правильный слой - домен и отдельный маскированный fallback, но не возврат к открытому Xray на голом IP.
