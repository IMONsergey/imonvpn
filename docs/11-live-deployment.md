# 11. Живое состояние деплоя

Документ фиксирует **реальную** конфигурацию текущего сервера, а не идеальную целевую архитектуру.

## Что реально запущено

- Провайдер: `AlphaVPS`
- Локация: `Sofia, Bulgaria`
- ОС: `Debian 12 (bookworm)`
- Основной стек:
  - `xray-reality` как **отдельный systemd service**
  - `nginx` как локальный fallback на `127.0.0.1:8081`
  - `amnezia-awg2` как отдельный `Docker`-контейнер для нативного `AmneziaWG`
- Причина перехода на standalone:
  - `x-ui` успешно ставился, но активный runtime `xray` стартовал с повреждённым списком клиентов
  - в боевом процессе часть inbound-ов имела `clients: null`, из-за чего валидные `VLESS Reality`-клиенты периодически получали HTTP fallback вместо туннеля

## Активные inbound-профили

### Рабочие

- `443/tcp` — `VLESS Reality`, основной профиль после тюнинга под `support.apple.com`
- `2053/tcp` — `VLESS Reality`, резервный профиль
- `2082/tcp` — `VLESS WS`, зарезервирован под дальнейший backup-сценарий
- `443/udp` — нативный `AmneziaWG` через контейнер `amnezia-awg2`

### Не рекомендовать клиентам как основной

- `8443/tcp` — профиль под Google-fronting сейчас ведёт себя нестабильно и не должен считаться primary

## Что уже проверено

- `xray-reality` поднимается после reboot
- `nginx` поднимается после reboot
- порты `443`, `2053`, `2082` слушаются после reboot
- клиентский трафик через локальный Xray-тест с Mac проходит успешно
- серверные outbound-запросы в интернет работают
- базовая защита от популярных IP-check сервисов включена
- `telegram.org`, `web.telegram.org`, `api.telegram.org` успешно отвечают через рабочий tuned-профиль `443/tcp`
- нативный `AmneziaWG` self-test через временный клиентский контейнер:
  - делает handshake
  - пропускает `1.1.1.1`
  - пропускает `telegram.org`
  - пропускает `api.telegram.org`

## Тюнинг, внесённый после первого запуска

- `443/tcp` переведён с Microsoft/Bing-паттерна на Apple-паттерн:
  - `dest = support.apple.com:443`
  - `serverName = support.apple.com`
- серверный DNS переведён с `DoH` на прямые `1.1.1.1`, `1.0.0.1`, `8.8.8.8` для уменьшения накладных расходов
- на inbound-ах включён `sniffing` для `http`, `tls`, `quic`

## Практический вывод по производительности

На контрольном локальном тесте через Xray-клиент:

- tuned `443/tcp` показал лучшую скорость, чем `2053/tcp`
- при этом tuned `443/tcp` успешно отработал телеграм-направления

Поэтому текущий рекомендуемый порядок такой:

1. `443/tcp` — основной
2. `2053/tcp` — резервный

## Разбор инцидента 2026-06-30

Проблема оказалась не в полном падении сервера, а в конфликте процессов:

- одновременно работали:
  - `standalone xray-reality`
  - старый `x-ui` со своим встроенным `xray`
- оба процесса слушали одни и те же порты:
  - `443`
  - `2053`
  - `8443`
  - `2082`

Критичный момент:

- у `x-ui` runtime-конфига на inbound-ах были `clients: null`
- из-за этого часть входящих подключений могла попадать не в рабочий `standalone xray-reality`, а в битый старый `x-ui xray`

Итог:

- с точки зрения пользователя VPN выглядел как "то грузится, то не грузится"
- это особенно похоже на проблему Amnezia/Hiddify, хотя корень был на сервере

Что сделали:

- остановили `x-ui`
- вручную замаскировали его systemd unit
- оставили на боевых портах только один процесс `xray-reality`

После этого:

- `telegram.org` снова успешно проходил через основной профиль `443`
- `api.telegram.org` снова успешно проходил через основной профиль `443`

Практический вывод:

- `x-ui` больше не должен участвовать в боевом трафике
- единственный допустимый прод-путь на этом сервере — `standalone xray-reality`

## Разбор инцидента Amnezia no-network

После починки `x-ui`-конфликта выяснилось, что у `AmneziaVPN` был ещё отдельный класс проблем:

1. На сервере сначала вообще не было нативного `AmneziaWG`.
2. После первой установки `AWG` контейнер поднимался, но его `start.sh` был загружен как шаблон без подстановки реальной подсети.
3. Из-за этого туннель делал handshake, но не создавал рабочий `MASQUERADE`/forwarding для выхода в интернет.
4. Дополнительно клиентские `.conf` пришлось перевести на `IPv4-only`, чтобы убрать лишний `::/0` маршрут на текущем single-VPS.

Итоговое исправление:

- установлен отдельный контейнер `amnezia-awg2`
- переведён на `UDP 443`
- `start.sh` переписан с реальной подсетью `10.8.1.0/24`
- клиентские `.conf` очищены от пустых `I2-I5`
- клиентские `.conf` переведены на `AllowedIPs = 0.0.0.0/0`

## Что ещё не завершено

- Домен и `Cloudflare` backup-слой
- Отдельный удобный клиентский путь для iPhone
- Ротация root-пароля и перевод SSH на ключи

Пункт про `AmneziaWG` теперь закрыт на текущем single-VPS:

- нативный протокол уже работает
- конфиги выпущены
- следующая стадия для него теперь не установка, а масштабирование и резервирование

## Практический вывод

На текущем этапе проект уже имеет **рабочий Reality VPN**, который можно импортировать в совместимые клиенты и использовать как стартовую боевую конфигурацию.

Следующее самое полезное усиление — добавить **домен** и вынести резервный профиль за `Cloudflare`, чтобы у пользователя был не один путь подключения, а минимум два разных по маскировке канала.
